# 三体迭代建造法 (Trinity Build)

## 概述

将一份蓝图文档变成一个完整、可运行、已测试的系统。核心原则：**先写文档，再写代码。模拟真实用户。修一个 bug 之前先防一类 bug。不做完端到端验证，不说做完了。**

## 执行流程

每一步完成后向用户报告进度，等待确认再进入下一步。

### Step 0: 确认蓝图文档

检查用户是否提供了文档。如果没有——引导用户写蓝图。蓝图至少包含：目录结构、数据模型、API 设计、实施阶段。

如果用户的输入是一段需求描述而非蓝图，用三源验证扩展它：
- **Industry**: 世界上最好的同类系统怎么做的？
- **Academic**: 这个问题的第一性原理是什么？
- **Reality**: 实际环境有什么限制？

有文档 → 进入 Step 1。

### Step 1: 文档翻译成代码

```
文档的每一行 → 一个函数/模块/端点
不是"参考文档写代码"，是"翻译文档成代码"
```

- 接口名、字段名、表名直接从文档取
- 不一致时**改代码，不改文档**。文档是真相源
- 翻译完毕后打开关键文件让用户确认

### Step 2: 模拟用户

```
打开浏览器/终端。像真实用户一样操作。完整走一遍。
```

- 不跳过任何步骤
- 记录每一个"不对"的瞬间
- 自动化测试抓不到静默失败（字段名不匹配不抛错，action type 无对应 case 不抛错）

### Step 3: 分类并建防线

每修一个 bug，先问：

> "这类问题会反复出现吗？"

**会** → 建自动化防线（linter、验证器、后处理器）
**不会** → 修完继续

防线示例：
- 接口合约 → 自动检测 action type/字段名 是否匹配
- 品牌/主题注入 → 确定性后处理，不依赖 LLM 遵守 prompt
- 结构性缺陷 → 正则匹配函数体（如 render 是否被调用）

### Step 4: 完整性检查

回到蓝图文档，逐一对照"说的"和"做的"：

| 文档说的 | 代码有的 | 状态 |
|---------|---------|------|
| ... | ... | ✅/❌/⚠️ |

缺口分级：
- **P0**: 系统跑不通
- **P1**: 主线断裂
- **P2**: 用户不爽

P0 → P2 顺序修复。

### Step 5: 端到端验证

```bash
# 全部服务启动
# curl 每个端点
# 浏览器走通每条路径
# 每个 URL 都 200，每个按钮都点过
```

### Step 6: 人工验收

列出验收清单，逐项打勾：
- [ ] 页面 A 正常
- [ ] 功能 B 可用
- [ ] 流程 C 走通

**机器验证告诉你"没 crash"。人眼验证告诉你"能用"。**

## 红线

- 代码和文档不一致 → 改代码
- "应该没问题" → 去验证
- 只修 bug 不问"会不会反复出现" → 先问
- 没走过端到端就说"做完了" → 走完再说

## 适用场景

| 适合 | 不适合 |
|------|--------|
| 从文档构建完整系统 | 5 分钟的小改动 |
| 已有项目加新功能 | 纯探索性原型 |
| 重构老系统 | 已有清晰 spec 的简单实现 |
