## 第二章：三体迭代——一套可以重复的建造方法

### 什么是三体迭代

"三体"是三个并行运作的验证流：

```
体1: 行业最佳实践（Industry）
     "世界上最强的 18 个系统是怎么做的？"
     → 研究 Firebase、Duolingo、Voucherify、淘宝千牛……

体2: 学术/理论基础（Academic）
     "这个问题的第一性原理是什么？"
     → 身份 = 设备指纹 + 渐进式升级
     → 经济 = 货币流入（QR grant）+ 货币流出（play cost）+ Sink（兑换消耗）

体3: 真实环境验证（Reality）
     "真的跑起来是什么样？"
     → 打开 HTML，真的点，真的玩
     → curl 每个端点，检查返回
     → 模拟用户完整路径
```

**三源交叉验证：** 行业说"用 CNAME"，学术说"CNAME 正在被浏览器封杀"，真实环境说"茶叶店配不了 DNS"。三个源打架 → 方案不成立 → 换方案。

### 三体迭代的循环

```
研究（Industry + Academic + Reality）
    ↓
设计（写文档，把结论定下来）
    ↓
建造（文档直接映射成代码）
    ↓
验证（模拟用户，真实使用）
    ↓
修复（从修一个 bug 到防一类 bug）
    ↓
完整性检查（回到文档，对比"说的"和"做的"）
    ↓
回到研究（发现的新问题 → 三源验证）
```

## 第四章：模拟用户——最强大的测试工具

### 不是自动化测试。是人真的去用。

写完代码后，第一步不是跑 `pytest`。是打开浏览器，输入 `http://localhost:8000`，**像一个真实的咖啡店老板一样去操作**。

### 第一次模拟：星巴克老板来定制游戏

**操作路径：**
1. 打开 Portal → 登录（老王咖啡账号）
2. 进入 Games → 描述业务："我是星巴克，卖咖啡和甜点"
3. 系统推荐了 Coffee Shop（0.95 分）→ 选择
4. 等待生成 → 打开游戏

**发现的问题（一次模拟，暴露 6 类 bug）：**

| 问题 | 用户看到的 | 根因 |
|------|----------|------|
| 品牌没生效 | 紫粉色背景，"星巴克绿在哪？" | CSS 变量 `--kix-brand` 设了但**无人引用** |
| 不知道点哪里 | "这些色块是什么意思？" | 纯色方块，无文字，无引导 |
| 点击没反应 | "点了 PLAY 什么都没发生" | 按钮发 `start_game`，handleAction 无此 case → `default: break` |
| 一碰就死 | "刚点一个棋子就 Game Over" | `checkGameOver` 用 `state.boardSize`（undefined）→ 循环不执行 → 默认返回 true |
| 分数显示空白 | "Moves 那里是空的" | render 读 `state.movesLeft`，initState 声明 `state.moves` |
| 其他按钮全坏 | "pause/settings/shop 都不工作" | **11 处** action type 与 switch case 不匹配 |

**这些 bug 的共同特征：代码没有 crash。静默失败。自动化测试抓不到。**

只有人真的去点、真的去玩，才会发现。

### 关键原则 #3：模拟用户不是"测一下"，是"完整走一遍"

不跳过任何步骤。不假设任何步骤"应该没问题"。从打开浏览器到完成操作，每一秒都在观察。

## 第五章：从修一个 bug 到防一类 bug

### 分类，然后建自动化防线

修完 12+ 处手动 bug 后，不是继续修下一个。是停下来问：

> "这一类问题会反复出现吗？"

**会。** 因为 Code-Soul 是 LLM 生成的代码。LLM 每次都可能写出不同的字段名、不同的 action type。今天是 `movesLeft`，明天可能是 `remainingMoves`。

### 防线一：brand_injector.py

**问题类：** LLM 生成代码时"忘记"品牌约束。prompt 里写了"星巴克绿 #00704A"，但生成的 HTML 是紫粉色。

**自动化方案：** 不等 LLM 遵守。代码生成完后，确定性替换：

```python
# 不依赖 LLM 遵守品牌约束
# 直接后处理：CSS 选择器覆盖颜色、HTML 标题替换品牌名、棋子 emoji 替换产品图标
post_process(html_path, brand_name="星巴克", primary_color="#00704A", products=["咖啡","甜点"])
```

### 防线二：contract_verifier.py

**问题类：** 事件监听发送的 action type 和 handleAction 的 switch case 不匹配。字段名在 initState、handleAction、render 三个地方不一致。

**自动化方案：** 正则提取 → 集合对比 → 已知映射表 → 自动替换：

```python
# 自动检测并修复
auto_fix_html(html)
# → 'action: start_game → navigate'
# → 'field: state.movesLeft → state.moves'
# → 'structure: added missing render() call in startGame'
```

### 关键原则 #4：修一个 bug 是修今天。防一类 bug 是修明天。

每修一个 bug，问自己："我能不能让这个 bug 再也出现不了？" 如果能，建防线。

## 第九章：你也能做到——三体迭代建造法的完整流程

### 方法速查

```
Step 0: 写蓝图文档
   ├── 研究（Industry + Academic + Reality 三源验证）
   ├── 目录结构（每个目录、每个文件、每个功能）
   ├── 数据模型（每张表、每个字段、每个关系）
   ├── API 设计（每个端点、输入、输出）
   ├── 实施阶段（每阶段明确的交付物）
   └── 参考系统（至少 5 个对标的真实系统）

Step 1: 文档翻译成代码
   ├── 文档的每一行 → 代码的一个函数/模块
   └── 不一致时改代码，文档是真相源

Step 2: 模拟用户
   ├── 打开浏览器，真的点、真的玩
   ├── 完整走一遍用户路径
   └── 记录每一个"不对"的瞬间

Step 3: 分类和建防线
   ├── 修一个 bug → 问"这类问题会反复出现吗？"
   ├── 会 → 建自动化防线
   └── 不会 → 修完继续

Step 4: 完整性检查
   ├── 回到蓝图文档
   ├── 逐一对照"说的"和"做的"
   ├── P0/P1/P2 分级
   └── 全部关闭

Step 5: 端到端验证
   ├── 全部服务启动
   ├── curl 每个端点
   └── 浏览器走通每条路径

Step 6: 人工验收
   ├── 真人使用
   ├── 验收清单逐项打勾
   └── 确认"能用"而不仅是"没 crash"
```

### 这个方法的适用场景

| 场景 | 适用性 |
|------|--------|
| 新产品从零构建 | ✅ 完整流程 |
| 已有产品加新功能 | ✅ 从 Step 0 开始，范围缩小 |
| 重构老系统 | ✅ 先写蓝图 → 再对照 |
| Bug 修复 | ✅ Step 3: 从修一个到防一类 |
| 技术选型 | ✅ Step 0: 三源验证 |

### 这个方法的限制

| 不适用 | 原因 |
|--------|------|
| 5 分钟的小改动 | 不需要蓝图文档 |
| 不确定的探索 | 先做原型，再写文档 |
| 已有清晰 spec 的实现 | 从 Step 1 直接开始 |

## 第十章：给不同角色的建议

### 产品经理

你的核心产出不是 PRD。是**可执行的项目计划**——包含目录结构、数据模型、API 设计。不是说"应该有一个推荐系统"，是说"推荐系统有三层：research → prefilter → LLM rank，每层的输入输出是 X"。

### 工程师

你的核心能力不是在 IDE 里写代码。是**把文档翻译成代码**。文档是你和 PM 之间的合约。文档写清楚了，你不需要猜。文档写得不清楚，先修文档再写代码。

### 设计师

你的核心价值不是画界面。是**在模拟用户阶段发现体验断裂**。打开 HTML，真的用。按钮该在哪、流程该走几步、用户不知道点哪——只有人眼能发现。

### 市场/运营

你的核心贡献在 Step 0。**哪些参考系统值得研究？** Firebase 怎么做的？Duolingo 怎么做的？对手怎么做的？这些信息直接决定系统设计。
